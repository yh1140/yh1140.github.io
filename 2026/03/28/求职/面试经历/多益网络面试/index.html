虚拟机部署和容器化部署的区别

  • 虚拟机部署是通过 Hypervisor 技术在物理硬件上模拟出完整的 Guest OS，具备极强的隔离性和安全性，但系统开销大、启动缓慢且资源利用率相对较低；
  • 而容器化部署（如 Docker）则是操作系统级的虚拟化，通过共享宿主机内核并利用 Namespace 和 Cgroups 实现进程级隔离，具有极致的轻量化、秒级启动和高度的环境一致性。

虚拟机部署和容器化部署与系统层面交互的细节？

  • 从底层执行逻辑来看，
  • 当我在虚拟机中执行一个命令（如 ls）时，
    • 该请求必须穿过 Guest OS 的完整内核栈，再由 Hypervisor（如 KVM）进行指令拦截与硬件地址翻译，最后才提交给物理机内核，这种多层封装导致了明显的 CPU 指令损耗和 IO 延迟；
  • 而在 Docker 容器中执行同样命令，它本质上只是宿主机上的一个受限进程，
    • 通过 execve 系统调用直接与宿主机内核交互，中间没有任何虚拟化转换层，仅靠 Namespace 机制在进程视图上做了一层“障眼法”隔离，并由 Cgroups 严格限制其资源配额，这种“零翻译”的特性使得容器在执行效率上几乎等同于原生进程，这也是我在 K8s 生产实践中，能够实现秒级扩容并支撑高并发业务的关键所在。

用户态和内核态的区别？

  • 用户态和内核态是操作系统为了保障系统稳定性与安全性而设计的两种 CPU 特权级别：
  • 用户态（Ring 3）特权较低，仅能访问受限的用户空间内存且无法直接操作硬件，
  • 而内核态（Ring 0）拥有最高特权，可执行所有指令并完全掌控 CPU、内存及外设资源。
  • 当应用程序需要执行 I/O 读写、网络通信或进程管理等涉及硬件的操作时，必须通过系统调用（System Call）引发特权等级切换，进入内核态执行。
  • 这种隔离机制能有效防止单个进程的逻辑错误导致整个系统发生 Kernel Panic（内核崩溃），但在运维调优场景中，由于状态切换涉及 CPU 寄存器保存、堆栈切换等上下文切换开销，我们会重点关注 context switch 等指标，以评估内核交互频率对高并发业务吞吐量的性能损耗。

容器之间的隔离如何实现的？

  • 容器本质上是宿主机上的一个特殊进程，
  • 其隔离性主要依托 Linux 内核的 Namespace（命名空间）、Cgroups（控制组） 以及 Layered File System（分层文件系统） 三大核心技术实现：
    • 首先，Namespace 负责“视图”隔离，通过 PID、Net、Mount、UTS、IPC 和 User 等维度让容器拥有独立的网络栈、进程空间和挂载点，实现环境解耦；
    • 其次，Cgroups 负责“资源”配额，精确限制容器对 CPU、内存、磁盘 I/O 等物理资源的使用，防止资源争抢导致的“噪声邻居”效应；
    • 最后，利用 chroot/pivot_root 配合 UnionFS 挂载镜像层，实现了文件系统的完全独立与一致。
  • 这种共享宿主机内核的“逻辑隔离”机制使得容器相比虚拟机（VM）更加轻量，但作为 SRE 视角，我也非常关注共享内核带来的安全风险，通常会结合 Seccomp、AppArmor 以及权限权能（Capabilities）等手段进一步加固内核边界。

  • TCP 协议作为传输层协议，其核心是面向字节流的有序可靠传输，本身并不感知应用层“请求”的概念，其处理方式主要由应用层协议及连接管理策略决定：
  • 在传统的短连接模式下，每个请求会分别触发独立的三次握手；若采用长连接（Keep-Alive），则取决于协议版本，HTTP/1.1 默认在同一链路上串行处理（Request-Response 模式），虽然支持 Pipelining 但由于 TCP 必须按序交付，仍会产生严重的队头阻塞（HoL Blocking）问题；
  • 而现代的 HTTP/2 协议通过引入二进制分帧层，在单一 TCP 连接上实现了多路复用（Multiplexing），允许两个请求的报文乱序交错传输并在对端重组，从而解决了并发效率瓶颈。在生产环境的运维实践中，我们通常需要结合内核的 tcp_max_syn_backlog 等参数优化连接排队机制，并利用 Nginx 的反向代理策略来合理分配长连接的生命周期与请求上限。

通信时的底层实现，cpu、存储等的细节

  • 通信的底层实现是一个深度协同硬件与内核的闭环过程：
  • 应用层数据首先通过系统调用从用户态拷贝至内核态的 Socket 缓冲区，由 CPU 介入执行协议栈的封装逻辑并进行校验和计算；
  • 随后，内核调度 DMA（直接存储器访问） 引擎将数据直接从内存传输至网卡（NIC）的硬件队列中，从而规避了 CPU 在大块数据搬运时的算力浪费；而在高并发场景下，CPU 频繁的中断处理和进程上下文切换会直接导致 L1/L2 缓存污染，因此我们在 SRE 实战中常通过 零拷贝（Zero-copy） 技术（如 sendfile）减少内存冗余拷贝，或利用 CPU 亲和性（Affinity） 绑定核心以减少 L3 缓存失效，确保数据在“内存-内核-网卡”路径上的极速流转。

8、简述交换机 集线器 路由器区别和作用？

  • 首先说HUB,也就是集线器 。它的作用可以简单的理解为将一些机器连接起来组成一个局域网。

  • 交换机 （又名交换式集线器）作用与集线器大体相同。但是两者在性能上有区别：集线器采用的式共享带宽的工作方式，而交换机是独享带宽。这样在机器很多或数据量很大时，两者将会有比较明显的。

  • 路由器 与以上两者有明显区别，它的作用在于连接不同的网段并且找到网络中数据传输最合适的路径 ，一般情况下个人用户需求不大。路由器是产生于交换机之后，就像交换机产生于集线器之后，所以路由器与交换机也有一定联系，并不是完全独立的两种设备。路由器主要克服了交换机不能路由转发数据包的不足。

  • 总的来说，路由器与交换机的主要区别体现在以下几个方面：

  • 工作层次不同

  • 最初的的交换机是工作在OSI／RM开放体系结构的数据链路层，也就是第二层，而路由器一开始就设计工作在OSI模型的网络层。由于交换机工作在OSI的第二层（数据链路层），所以它的工作原理比较简单，而路由器工作在OSI的第三层（网络层），可以得到更多的协议信息，路由器可以做出更加智能的转发决策。

  • 数据转发所依据的对象不同

  • 交换机是利用物理地址或者说MAC地址来确定转发数据的目的地址。而路由器则是利用不同网络的ID号（即IP地址）来确定数据转发的地址。IP地址是在软件中实现的，描述的是设备所在的网络，有时这些第三层的地址也称为协议地址或者网络地址。MAC地址通常是硬件自带的，由网卡生产商来分配的，而且已经固化到了网卡中去，一般来说是不可更改的。而IP地址则通常由网络管理员或系统自动分配。

  • 传统的交换机只能分割冲突域，不能分割广播域；而路由器可以分割广播域，由交换机连接的网段仍属于同一个广播域，广播数据包会在交换机连接的所有网段上传播，在某些情况下会导致通信拥挤和安全漏洞。连接到路由器上的网段会被分配成不同的广播域，广播数据不会穿过路由器。虽然第三层以上交换机具有VLAN功能，也可以分割广播域，但是各子广播域之间是不能通信交流的，它们之间的交流仍然需要路由器。

  • 路由器提供了防火墙的服务，路由器仅仅转发特定地址的数据包，不传送不支持路由协议的数据包传送和未知目标网络数据包的传送，从而可以防止广播风暴。

  • 交换机一般用于LAN-WAN的连接，交换机归于网桥，是数据链路层的设备，有些交换机也可实现第三层的交换。 路由器用于WAN-WAN之间的连接，可以解决异性网络之间转发分组，作用于网络层。他们只是从一条线路上接受输入分组，然后向另一条线路转发。这两条线路可能分属于不同的网络，并采用不同协议。相比较而言，路由器的功能较交换机要强大，但速度相对也慢，价格昂贵，第三层交换机既有交换机线速转发报文能力，又有路由器良好的控制功能，因此得以广泛应用。

20、介绍一下 ACL和 NAT？NAT有几种方式？

  • 访问控制列表（ ACL）是应用在路由器接口的指令列表（规则） ，用来告诉路由器 哪些数据包可以接收转发，哪些数据包需要拒绝；
  • ACL 的工作原理 ：读取第三层及第四 层包头中的信息，根据预先定义好的规则对包进行过滤；
  • 使用 ACL 实现网络控制：实现 访问控制列表的核心技术是包过滤；
  • ACL 的两种基本类型（标准访问控制列表；扩展访 问控制列表）
  • NAT：改变 IP包头使目的地址，源地址或两个地址在包头中被不同地址替换。
  • NAT有的方式：静态 NAT、动态 NAT、PAT

22、NAT 的原理及优缺点

原理：

  • 转换内部地址；
  • 转换外部地址；
  • PAT 端口地址转换；
  • 解决地址重叠问题；

优点：

  • 节省 IP 地址 ,能够处理地址重复的情况；
  • 增加了灵活性 , 消除了地址重新编号 ,隐藏了内部 IP 地址；

缺点：

108.为什么要使用OSPF多区域？

1、在大型网络中，网络结构的变化是时常发生的，因此OSPF路由器就会经常运行SPF算法来重新计算路由信息，大量消耗路由器的CPU和内存资源。
2、在OSPF网络中，随着多条路径的增加，路由表变得越来越庞大，每一次路径的改变都使路由器不得不花大量的时间和资源去重新计算路由表，路由器越来越低效。
3、包含完整网络结构的链路状态数据库也会越来越大，这将有可能使路由器的CPU和内存资源彻底耗尽，从而导致路由器的崩溃。
然后，通过OSPF划分区域以便引出OSPF多区域划分的原因

改善网络的可扩展性
快速收敛

由于流量整形使用缓存来保存违规流量，所以可以减小丢包率，避免了报文重发，但是使用缓存将导致数据转发时增加时延。流量整形一般应用在整体流量较小，但有时出现突发流量的环境。

GTS只能应用在output方向，CAR可以应用在两个方向。

将需要转发的数据按照调度顺序进行转发。

在出队的时候，WFQ按照流的优先级来分配出口的带宽，优先级越小得到的带宽越小，
优先级越大所得到的带宽越大

CAR技术多应用于接入层对接入用户经行流量限制。)

多益网络运维工程师面试会问的问题

多益网络网络运维工程师专业面试通常涵盖以下核心领域，结合技术知识、实战能力和个人素质进行考察：

一、网络基础知识

  • TCP/IP 协议：

    TCP 三次握手 / 四次挥手流程

    （必考）、正向代理与反向代理区别、DNS 解析流程、NAT 原理及应用。

  • 网络设备与技术：VLAN 作用、STP 协议功能、ACL 配置与应用、SD-WAN 与传统 MPLS 对比优势。

  • DHCP 服务：地址分配流程及客户端获取 IP 失败排查（如服务未启动、地址池耗尽、DHCP 中继缺失等）。

二、故障排查与实战能力

  • 广播风暴：通过设备指示灯高频闪烁、CPU 利用率异常等现象判断，结合 STP 协议或环路检测工具定位解决。
  • 物理链路故障：使用 ping、traceroute 命令及端口状态检查（如 show interfaces）验证连通性。
  • 网络环路：观察设备端口 “齐闪齐灭”、高 CPU 占用，通过 show process cpu 命令排查。

三、系统与工具应用

  • Linux 系统管理：基础命令（如进程管理、日志分析）、服务配置（如 Nginx、防火墙）。
  • 监控工具：Zabbix、Prometheus 等工具的使用场景及日志分析方法。
  • 备份策略：网络设备配置备份（如定期自动备份、版本控制）及恢复流程。

四、综合技能与个人素质

  • 项目经验：过往运维项目中的职责、技术难点及解决方案（面试核心，占比 24.7%）。
  • 脚本开发：Python 自动化脚本编写（如批量配置设备、日志处理）。
  • 个人认知：优缺点分析（建议结合学习能力、实战经验不足等客观表述）。

建议结合多益网络业务场景（如游戏服务器运维、高并发网络架构），重点准备网络稳定性保障、自动化运维工具应用等相关案例，突出技术深度与问题解决能力。
