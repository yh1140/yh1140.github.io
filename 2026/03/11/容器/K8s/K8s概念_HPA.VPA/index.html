K8s概念_HPA.VPA

发表于2026-03-11|更新于2026-03-11|K8s
|浏览量:

在运维世界里，应对流量压力有两条路：加人（横向） 或 增强单兵作战能力（纵向）。

核心概念

特性 HPA (Horizontal Pod Autoscaler) VPA (Vertical Pod Autoscaler)
伸缩方向 横向（增加/减少 Pod 副本数） 纵向（增大/减小单个 Pod 的 CPU/内存）
原理 复制更多的实例来分担压力 调整容器的 requests 和 limits
适用场景 无状态服务（Web 接口、微服务） 有状态服务、数据库、单体应用
副作用 可能会受到存储或数据库连接数限制 传统模式下需重启 Pod（1.35版本支持原地更新）

HPA

通过定期（默认 15s）查询 metrics-server 获取 Pod 的资源利用率。

核心工作流

  • 监控：HPA 控制器查询 metrics-server。

  • 计算：根据公式计算期望副本数：

    $$期望副本数 = \lceil 当前副本数 \times \frac{当前指标值}{期望指标值} \rceil$$

  • 执行：修改 Deployment/StatefulSet 的 replicas 字段。

生命周期

当你执行 kubectl apply -f hpa.yaml 后，系统内部发生了以下过程

①控制器激活

HPA 控制器（属于 kube-controller-manager 的一部分）开始运行。它会根据 scaleTargetRef 找到你的 Deployment。

  • 故障点：如果 Deployment 名字写错，HPA 状态会显示 TargetNotFound。

②指标获取(Metrics Collection)

HPA 控制器每隔 15 秒（默认）请求 Metrics Server。

  • 底层链路：HPA Controller -> API Server -> Metrics API (Aggregation Layer) -> Metrics Server -> Kubelet (Summary API)。

  • 故障点：如果 kubectl top pods 没数据，HPA 永远不会生效。

③算法计算

使用公式：期望副本数 = ceil[当前副本数 * (当前指标 / 期望指标)]。

  • 注意：如果计算结果波动在 10% 以内（默认容忍度），HPA 不会执行任何动作，这是为了系统稳定。

④执行扩缩容

HPA 通过 Scale 子资源接口，修改 Deployment 的 replicas 字段。

  • 验证方法：kubectl describe hpa php-apache-hpa。在底部的 Events 栏可以看到 SuccessfulRescale 的记录。

VPA

VPA 的核心价值在于：解决研发人员乱填 resources 的问题。它能根据历史负载，自动给出一个合理的资源建议。

VPA三个组件

  • Recommender：后台观察历史数据，计算建议值（Target）。

  • Updater：发现 Pod 的资源偏离建议值太大时，负责杀掉 Pod（如果开启了 Auto 模式）。

  • Admission Controller：在 Pod 重建时，拦截请求并将 requests 修改为建议值。

HPA创建

命令创建

速度快，适合临时扩容

  • autoscale: 告知 kubectl 我要创建一个 HPA 对象。

  • deployment php-apache: 指定要监控的目标。

  • --cpu-percent=50: 目标 CPU 使用率阈值。

  • --min=1 --max=10: 副本数的上下限。
    这条命令背后其实是 K8s 帮你自动生成了一个 v1 版本的 HPA。它功能较弱，通常只能根据 CPU 缩放。

YAML创建

运维标准做法，支持 多指标监控 和 扩缩容策略（Behavior） 的精细化控制
