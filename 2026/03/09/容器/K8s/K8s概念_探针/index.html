K8s概念_探针

发表于2026-03-09|更新于2026-03-11|K8s
|浏览量:

摘要：

定义与基本概念

探针是 Kubernetes 监控容器运行状态的“传感器”，它由 Kubelet 定期执行，根据探测结果决定是否重启容器或切断流量，从而实现服务的自动化健康管理。

探针分类

  • liveness探针：影响的是单个容器，如果检查失败，将杀死容器，根据pod的restartPolicy来操作。
  • readiness探针：影响的是整个pod，即如果pod中有多个容器，只要有一个容器的readiness探针诊断失败，那么整个pod都会处于unready状态。
  • startup探针：指示容器中的应用是否已经启动。如果提供了启动探针(startup probe)，则禁用所有其他探针，直到它成功为止。如果启动探针失败，kubelet 将杀死容器，容器服从其重启策略进行重启。如果容器没有提供启动探针，则默认状态为成功Success。

探针的参数

参数名 YAML 字段 默认值 定义与运维意义
初始延迟 initialDelaySeconds 0 容器启动后等多久才开始第一次探测。给应用留出启动时间。
探测间隔 periodSeconds 10 探测的频率。越小感知故障越快，但对容器负载也越高。
超时时间 timeoutSeconds 1 等待响应的时间。如果超过这个时间没返回，就算探测失败。
失败阈值 failureThreshold 3 连续失败几次才真正判定为失败（重启容器或剔除流量）。
成功阈值 successThreshold 1 连续成功几次才判定为恢复正常（通常保持默认值 1）。
终止宽限期 terminationGracePeriodSeconds 30 存活探针失败触发重启时，给容器留出的优雅退出时间。
  • initialDelaySeconds：容器启动后第一次执行探测是需要等待多少秒。
  • periodSeconds：执行探测的频率。默认是10秒，最小1秒。
  • timeoutSeconds：探测超时时间。默认1秒，最小1秒。
  • successThreshold：探测失败后，最少连续探测成功多少次才被认定为成功。默认是1。对于liveness必须是1。最小值是1。
  • failureThreshold：探测成功后，最少连续探测失败多少次才被认定为失败。默认是3。最小值是1。

探针的探测结果

  • Success：Container通过了检查。
  • Failure：Container未通过检查。
  • Unknown：未能执行检查，因此不采取任何措施。

三种探针

存活探针-Liveness Probe

必要性

在一个服务中，是存在进程在运行，但服务其实已经挂掉了，表现为端口监听失败、http请求失败等。所以需要存活探针

原理

启动探针-Startup Probe

Startup探针如果失败就不睡启动其它探针了

原理

探针的工作原理

执行主体

探针的执行者不是 Master 节点，而是 Kubelet。

  • 本地化执行：每个 Node 节点上的 Kubelet 进程内部都有一个 Probe Manager。它根据 Pod Spec 中的配置，定时发起探测动作。
  • 物理链路：Kubelet 直接与宿主机上的容器网络或 Namespace 通信。
    • 如果是 HTTP/TCP：Kubelet 直接访问容器的 IP 和端口。
    • 如果是 Exec：Kubelet 通过 CRI（Container Runtime Interface） 调用 containerd，在目标容器内执行命令。

循环检测

探针并不是执行一次就结束了，它是一个有状态的计数器。
计数器机制：
Kubelet 会在内存中为每个探针记录两个计数器：

  1. 连续成功计数 (Success Count)
  2. 连续失败计数 (Failure Count)
    状态转换原理：
  • 从正常到异常：只有当 Failure Count 连续达到 failureThreshold（失败阈值）时，状态才会真正改变。
  • 从异常到正常：只有当 Success Count 连续达到 successThreshold（成功阈值）时，才认为恢复。

    注意：这种“连续计数”机制是为了容错。它可以防止因为一次偶发的网络抖动（Flapping）导致容器被误杀。

探针联动

Pod 启动后，三类探针在时间轴上的协作逻辑：

①启动保护（Startup Probe）

  • 锁死机制：在启动探针 成功/彻底失败 之前，Liveness 和 Readiness 探针都会被强行挂起（停用）。
  • 目的：保护慢启动应用，防止在初始化时被 Liveness 重启。

② 平滑上线（Readiness Probe）

  • 原理：一旦启动探针通过，Readiness 开始高频工作。
  • 联动 Service：
    • 探测失败 $\rightarrow$ Kubelet 通知 API Server $\rightarrow$ Endpoint Controller 将该 Pod IP 从 Service 的 Endpoints 列表中移除。
    • 结果：外部流量（如经过 Ingress 或 Service 的流量）不再流向该 Pod。

③持续监控（Liveness Probe）

  • 原理：与 Readiness 并行运行，但不影响流量分配。
  • 联动自愈：
    • 探测失败 $\rightarrow$ Kubelet 直接调用 Containerd 停止容器 $\rightarrow$ 根据 restartPolicy 重建新容器。

四种探测方法

每种探针，每个阶段，都可以使用这四种探测方法

exec

  • 通用：所有类型的pod都可以用这种方法
  • 原理：进入pod内部执行命令，可以执行->pod是好的

httpget

通过http协议访问web服务，可以访问->pod是好的

tcpSocket

访问ip+端口，可以访问->pod是好的

grpc

调用功能，有响应->pod是好的
