K8s实战_探针

发表于2026-03-09|更新于2026-03-09|K8s
|浏览量:

实验环境

实验目标

  • 观察 Startup Probe 如何保护慢启动应用。

  • 验证 Readiness Probe 失败时，Service 如何自动切断流量（不重启容器）。

  • 验证 Liveness Probe 失败时，Kubelet 如何强制重启容器。

脚本准备

容器模拟逻辑

模拟三个阶段

  • 启动中：前 20 秒，文件不存在（此时 startupProbe 会失败）。
  • 正常运行：20 秒后，创建 /tmp/started（此时 startupProbe 成功，接力给后续探针）。
  • 状态模拟：通过检查是否存在 /tmp/overload 或 /tmp/deadlock 来改变探测结果。

编辑advanced-probe-lab.yaml

实操

启动&接力

模拟高负载(流量切断)

进入容器创建一个’过载‘标志文件

观察service

  • 发现 Endpoints 列表变为空，或者该 Pod 的 IP 被移除了。
  • 执行 kubectl get pods：Ready 变为 0/1，但 RESTARTS 为 0
  • 结论：应用只是慢了，没死。K8s 停止分配流量，保护应用不被压垮。
    恢复

发现流量自动恢复（Ready 1/1）

模拟死锁(自动重启)

触发崩溃

  • 观察：等约 10-20 秒。
    • 执行 kubectl get pods。
    • 你会看到 RESTARTS 计数变为 1！
  • 原理：livenessProbe 发现 exit 1，判定容器已不可用，Kubelet 强制杀死并重启容器。由于重启后是新容器，/tmp/deadlock 文件消失，Pod 重新恢复健康

总结

实验操作 触发探针 预期现象 运维意义
刚部署前 20s startup Ready 0/1, 0 Restarts 保护冷启动，防止启动慢被误杀
touch /tmp/overload readiness Ready 0/1, 0 Restarts 业务抖动时切断流量，实现平滑降级
touch /tmp/deadlock liveness Ready 0/1 -> 1/1, Restarts+1 核心逻辑卡死时自动拉起，实现自愈

Docker概念_命名空间
Docker 并不是一种全新的虚拟化技术，它本质上是一个利用 Linux 内核特性实现的隔离进程。 Docker 之所以能让容器看起来像一个独立的操作系统，全靠 Namespaces（资源隔离） 和 Cgroups（资源限制） 这两项核心技术。 基本定义命名空间（Namespaces） 是 Linux 内核的一个特性，用于对内核资源进行物理隔离。 通俗解释：想象你在一栋办公楼里。命名空间就像是给每个团队分配了一个独立的办公室。虽然大家都在同一栋楼（同一个 Linux 内核）里，但 A 团队看不见 B 团队的桌面，也听不见他们的电话。 运维意义：它保证了容器 A 里的进程编号、网络配置、挂载点等，不会与容器 B 或宿主机发生冲突。 七大命名空间PID Namespace-进程隔离NET Namespace-网络隔离MNT Namdespace-挂载点隔离UTS Namdespace-主机名隔离IPC Namdespace-进程间通信隔离USER Namespace-用户隔离CGROUP Namdespace
