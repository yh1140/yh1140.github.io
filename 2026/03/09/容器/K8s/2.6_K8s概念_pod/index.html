Pod的启动工作流

指令到达阶段-master节点

  • 用户请求：你通过 kubectl 发送 YAML 声明。

  • API Server 校验：API Server 接收请求，验证你的身份（RBAC）和 YAML 格式是否正确。

  • 持久化存储：校验通过后，API Server 将 Pod 的元数据（Metadata）写入 Etcd 数据库。此时 Pod 的状态是 Pending。

控制与调度阶段-master节点

  • 控制器检查 (Controller Manager)：Deployment 控制器发现预期的 Pod 副本数与实际不符，于是请求 API Server 创建一个新的 Pod 对象。

  • 筛选节点 (Scheduler)：

    • 预选 (Predicates)：排除掉资源不足（CPU/内存不够）或标签不匹配的节点。

    • 优选 (Priorities)：在剩下的节点中打分，选出最合适的。

  • 绑定 (Binding)：Scheduler 将选中的节点名称（如 node1）写入 Pod 的 spec.nodeName 字段，并更新回 API Server。

接力到物理节点-node节点

  • 监听 (Kubelet)：node1 上的 Kubelet 通过 Watch 机制发现有一个 Pod 分配给了自己。

  • 准备沙盒 (CRI)：Kubelet 调用容器运行时接口（CRI，在你的环境是 containerd）。

    • 创建 Pause 容器：containerd 首先启动 pause 容器，初始化 Network Namespace。这是 Pod 获得集群 IP 的时刻。

容器运行阶段-node节点

  • 拉取镜像：Kubelet 指挥 containerd 去你的镜像站（渡渡鸟）拉取业务镜像（如 Nginx）。

  • 启动业务容器：镜像拉取成功后，业务容器启动，并“加入”到 Pause 容器的网络空间中。

  • 状态回传：Kubelet 将容器的运行状态反馈给 API Server，存入 Etcd。此时 Pod 变为 Running。
