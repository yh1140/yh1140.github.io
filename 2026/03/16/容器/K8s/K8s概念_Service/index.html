K8s概念_Service

发表于2026-03-16|更新于2026-03-16|K8s
|浏览量:

在 K8s 中，Pod 是有生命周期的（销毁、重启、漂移…），这意味着 Pod 的 IP 是动态且不可靠的。

Service（简称 Svc） 的出现就是为了解决这个问题。它定义了一组 Pod 的逻辑集合，并为这组 Pod 提供一个固定的访问入口（IP 或域名）

  • 服务发现：无论Pod 的 IP 怎么变，客户端只需要访问 Service 的名称或其虚拟 IP
  • 负载均衡：当后端有多个副本（Replicas）时，Service 会自动将流量均匀地分发给健康的 Pod

三个组件

要理解 Service 的工作原理，必须看清这三个组件的关联

Service (SVC)

  • 定义访问规则和虚拟ip
  • 关注 selector 是否匹配， type 是否选对

Endpoints（EP）

  • 关键中间层，记录后端健康的 Pod IP 列表
  • 如果访问不通，先看 kubectl get ep 如果这里没有 IP 说明标签选错了

EndpointSlice

  • k8s 1.21+ 引入的高级版EP
  • 针对大规模集群（上千pod）的性能优化，K8s1.35默认推荐

三个端口

在编写 YAML 时，要定义三个端口

port

  • Service 在集群内部暴露的端口
  • 集群内其他Pod访问这个svc时用的端口

targetPort

  • 后端 Pod 容器里程序真实监听的端口

nodePort

  • 仅在 NodePort 类型下有效
  • 在每个物理节点上开启一个端口，供集群外部访问

四种Service类型

ClusterIP（默认）

  • 仅分配一个内部集群可达的虚拟IP

NodePort

  • 在每个节点上开启一个物理端口（30000-32767）

LoadBalancer

  • 请求云厂商创建一个负载均衡器
  • 生产环境的的公网入口
  • 会产生额外费用，需要云环境支持

Headless Service

-

ExternalName

  • 不通过标签选择器
  • 不分配虚拟ip
  • 通过DNSCNAME记录工作
  • 将集群外部的一个域名（比如阿里云的 RDS 地址或 GitHub 的 API 地址）映射到集群内部的一个 Service 名上

定义Service（不写Selector）

Docker概念_命名空间
Docker 并不是一种全新的虚拟化技术，它本质上是一个利用 Linux 内核特性实现的隔离进程。 Docker 之所以能让容器看起来像一个独立的操作系统，全靠 Namespaces（资源隔离） 和 Cgroups（资源限制） 这两项核心技术。 基本定义命名空间（Namespaces） 是 Linux 内核的一个特性，用于对内核资源进行物理隔离。 通俗解释：想象你在一栋办公楼里。命名空间就像是给每个团队分配了一个独立的办公室。虽然大家都在同一栋楼（同一个 Linux 内核）里，但 A 团队看不见 B 团队的桌面，也听不见他们的电话。 运维意义：它保证了容器 A 里的进程编号、网络配置、挂载点等，不会与容器 B 或宿主机发生冲突。 七大命名空间PID Namespace-进程隔离NET Namespace-网络隔离MNT Namdespace-挂载点隔离UTS Namdespace-主机名隔离IPC Namdespace-进程间通信隔离USER Namespace-用户隔离CGROUP Namdespace
