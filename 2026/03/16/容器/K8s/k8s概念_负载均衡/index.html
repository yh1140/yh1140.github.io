k8s概念_负载均衡

发表于2026-03-16|更新于2026-03-16|K8s
|浏览量:

kube-proxy:创建服务的时候，会创建负载均衡器，实现流量均衡的导流到 Pod 上
底层实现通过这些组件

  • Userspace：最原始的代理，流量直接进入kube-proxy进程内部，由进程在用户态转发。
  • iptables：默认模式，修改 Linux 内核的 iptables 规则，让内核在看到特定流量时直接进行 NAT（网络地址转换） 重定向
  • IPVS：高性能，利用 Linux 内核的 LVS (IP Virtual Server)。它专门为负载均衡设计，使用哈希表（Hash Table）来存储转发规则，用空间换时间

iptables

iptables 是 Linux 内核中 Netfilter 框架的用户态工具。在 K8s 中，它利用 NAT（网络地址转换） 规则将发往 Service VIP 的流量重定向到 Pod IP。

工作原理

iptables 的核心是链（Chains）。当流量进入节点时，它会按顺序匹配一条条规则：

  1. KUBE-SERVICES：入口，识别流量是否属于某个 Service。

  2. KUBE-SVC-XXX：对应具体的 Service，决定分发给哪个后端。

  3. KUBE-SEP-XXX：对应具体的 Endpoint（Pod），执行最终的 DNAT。

缺陷

  • 线性搜索：iptables 的规则是按顺序一条条往下“撞”的。如果你的集群有 5000 个 Service，每个 Service 有 3 个 Pod，那规则可能多达上万条。流量每经过一次都要从头匹配，延迟会随 Service 数量线性增长
  • 不可增量更新：每次修改规则（比如新起一个 Pod），都要先把整个表的规则导出来，改完了再塞回去。在超大规模集群下，这会导致内核 CPU 瞬间飙升。

IPVS

IPVS（IP Virtual Server）建立在 Netfilter 之上，采用了更高效的数据结构——哈希表（Hash Table）。

工作原理

与 iptables 逐条匹配规则不同，IPVS 将 Service 信息存入内核的哈希表中。

  • $O(1)$ 查找：无论你有 10 个 Service 还是 10000 个 Service，流量进入后通过哈希计算，瞬间就能找到目标 Pod。

  • 真正的负载均衡：iptables 的“负载均衡”纯靠随机概率；而 IPVS 支持多种专业算法：

    • rr (轮询)

    • lc (最小连接数)

    • dh (目标地址哈希)

Netfilter

Netfilter 是 Linux 内核中的一个框架，它在网络协议栈的各个关键点（Hooks）设置了“关卡”。任何进出服务器的数据包都必须经过这些关卡。

Netfilter 定义了 5 个核心挂钩（Hooks）：

五个核心挂钩

  1. PREROUTING: 数据包刚进入网卡。

  2. INPUT: 数据包目的地是本机。

  3. FORWARD: 数据包需要转发。

  4. OUTPUT: 本机产生的向外发出的数据包。

  5. POSTROUTING: 数据包即将离开网卡。

iptables

iptables 是完全基于 Netfilter 钩子构建的。它在这些钩子上挂载了大量的链（Chains）和规则（Rules）。

  • 工作方式：当数据包流经 Netfilter 的 Hook 时，iptables 会强迫数据包去匹配它定义的长长的规则链。

  • 局限性：因为它强依赖于 Netfilter 的链式逻辑，规则越多，数据包跑完这条“流水线”的时间就越长。

IPVS

IPVS 虽然也利用 Netfilter 的钩子，但它不走寻常路。

  • 工作方式：IPVS 主要挂载在 Netfilter 的 INPUT、FORWARD 和 POSTROUTING 钩子上。但是，一旦数据包进入 IPVS 的管辖范围，它就会跳过 iptables 那套繁琐的链式扫描，直接通过**哈希表（Hash Table）**快速定位目标。

  • 关系：IPVS 并不取代 Netfilter，它只是把 Netfilter 当作一个拦截流量的“触发器”，真正的负载均衡和转发逻辑是在 IPVS 自己的内核模块中高效完成的。

kube-proxy工作流程

监听->转换->写入

“简单来说，kube-proxy 在集群里扮演的是一个**‘规则同步员’的角色。它时刻盯着 API Server，一旦发现 Service 或 Pod 的 IP 变了，它就立马在每个节点的内核里更新一套‘交通调度表’（即 iptables 或 IPVS 规则）。最关键的一点是，它并不直接处理流量，而是把规则写好后就退到幕后，由 Linux 内核根据这些规则，在收到数据包的一瞬间将其‘瞬移’（DNAT）到正确的后端 Pod。所以，它是一个控制平面**在数据节点的延伸，负责把静态的配置变成动态的内核转发动力。”
