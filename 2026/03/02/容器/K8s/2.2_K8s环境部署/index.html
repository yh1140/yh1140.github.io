K8s环境部署

发表于2026-03-02|更新于2026-03-02|K8s
|浏览量:

摘要 :本文主要记录在

1.版本演进

从 v1.24 版本开始，K8s 正式移除了 Dockershim。这意味着 Docker 不再是默认的容器运行时。

  • v1.23 (旧架构)：k8s->docker->containerd->runc->内核lxc->容器
  • v1.24+ (新架构)：k8s->containerd->内核lxc->容器

2.集群部署的四种方法

2.1 kubeadm

K8s 社区官方推荐的工具，也是目前最通用的部署方式
手动准备操作系统、安装容器运行时（如 Containerd），然后通过 kubeadm init 和 kubeadm join 命令组建集群
优点 ： 流程标准化，能让你清晰理解证书管理、组件运行方式
缺点 ： 不包含高可用（HA）负载均衡器的配置、不包含监控和日志

2.2 自动化安装

使用 Ansible 脚本，将数以百计的手动操作封装成自动化剧本。

  • kubespray : K8s社区官方维护的ansible脚本，支持多个linux版本
  • kubeasz : 国内非常流行的开源项目，针对国内网络环境（镜像加速）做了大量优化，且内置了高可用方案和监控。
    优点 ： 一键安装，集成了高可用、网络插件（Calico/Cilium）、核心插件（CoreDNS/Dashboard）。

2.3 轻量安装工具

这类工具将 K8s 的所有组件打包成极小的二进制文件，甚至一键起停

  • K3s ：rancher出品，将控制平面打包进一个100MB左右的二进制文件，内存小号极低
  • MicroK8s ： 由 Canonical (Ubuntu 母公司) 开发，支持 Snap 一键安装。
  • Minikube ： 专门用于在本地开发环境中运行单节点 K8s。
    优点 ： 安装迅速，自动配置组件（包括存储和网络）
    缺点 ： 内部组件被替换 （Etcd -> SQLIte）

2.4 二进制手动安装

不借助任何自动化工具，手动下载 kube-apiserver、etcd、kubelet 等所有二进制文件

  • 手动生成CA证书
  • 手动缩写Systemd服务配置文件
  • 手动配置网络转发与IPVS
    优点 ：对集群有绝对的控制力，哪里坏了看一眼日志就能精准定位

2.5 其他方法

  • 云服务商托管部署
  • 容器镜像部署

3 部署思路

搭建一个分布式操作系统

3.1 第一层：地基（操作系统与网络预处理）

  • 身份识别：每台机器的hostname和ip要在/etc/hosts中登记
  • swap：必须关掉，磁盘交换会拖慢响应速度
  • 防火墙/selinux：关闭，避免 它们拦截 Pod 之间的通信
  • 通信补丁 （Bridge） ： Linux 默认不让网桥流量经过 iptables，但 K8s 必须让它经过，所以要加载 br_netfilter 模块并设置 net.bridge.bridge-nf-call-iptables = 1。

3.2 第二层：动力源（容器运行时）

K8s 只是一个“调度员”，它不直接管理容器，而是命令“工人”去做。

  • 选择工人：以前直接选 Docker。但在 v1.24+ 之后，K8s 移除了原生支持。
  • 中间人（CRI）：现在主流做法是使用 Containerd 或 Docker + cri-dockerd。
  • 如果你用 Docker，就必须装 cri-dockerd。它相当于一个翻译官，把 K8s 的命令翻译给 Docker 听。
  • 驱动一致性：必须让工人和调度员都使用同一种管理方式（通常是 systemd）。

3.3 第三层：骨架 （K8s 三大件安装）

  • kubelet：每个节点上的“工头”，负责接收 Master 指令并操作容器。
  • kubeadm：安装工具，相当于“自动化施工队长”，能帮你自动配置复杂的证书和核心组件。
  • kubectl：给集群下指令的“控制台”。

此时的状态：骨架搭好了，但还没通电，没联网。
