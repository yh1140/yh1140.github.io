普罗米修斯和zabbix的差异

维度 Prometheus (普罗米修斯) Zabbix
数据模型 多维时序数据 (Labels 标签) 树状层次结构 (Host-Item)
获取方式 Pull (拉取) 模式为主 Push (推/主动) 模式为主
存储引擎 自研 TSDB (高性能时序库) 传统关系型数据库 (MySQL/PgSQL)
服务发现 强 (自动发现 K8s 资源) 弱 (需手动或脚本注册)
适用场景 容器化、微服务、动态伸缩环境 物理机、网络设备、传统稳定架构

解释一下SOP是什么

  • SOP 的全称是 Standard Operating Procedure，即 标准作业程序
  • 一个SOP需要包括
    • 前置检查
    • 核心步骤
    • 验证环节
    • 回滚预案
    • 风险提示
  • 在执行任何生产操作之前，需要在测试环境验证流程并形成标准的SOP。
  • SOP不仅要包含正向操作步骤，一定要有验证和可执行的回滚方案

VM模板制作是怎么做的

  • VM模板是一个虚拟机的黄金镜像，包含以下内容的模板
    • 操作系统
    • 基础软件配置
    • 安全补丁
    • 优化参数
  • 制作一个VM模板的步骤如下
    • 基础安装优化
      • 最小化安装
      • 打系统补丁
      • 标准化配置
        • NTP自动对时
        • 监控，日志
    • 去标识化
      • 网络规则
      • SSH主机密钥
      • 日志历史
      • ID
    • 封装-设置虚拟机为模板（只读克隆，不可直接启动）

解释一下CMDB

  • CMDB (Configuration Management Database)，即 配置管理数据库。
    • 运维系统的“元数据中心”和“唯一信任源 (Single Source of Truth)”。
    • 简单来说，它记录了公司里所有 IT 资源的“户口本”。
  • CMDB实质
    • 唯一信任源
      • 收集运维数据，建立权威的，唯一的数据标准
    • 关系映射引擎
      • 水平关系-MySQL主从关系，MGR节点关系
      • 垂直关系-Pod运行在VM上，VM运行在宿主机上，宿主机插在交换机A口上
    • 运维自动化逻辑开关
      • CMDB是所有自动化脚本的入参
      • 自动化脚本执行第一步是查询CMDB获得参数
  • CMDB储存内容
    • CI
      • 服务器、虚拟机、数据库实例、交换机、甚至是一个 App。
    • 属性
      • IP 地址、CPU/内存规格、操作系统版本、所属业务组、负责人
    • 关系-CMDB灵魂
      • MySQL运行在哪台虚拟机上
      • 业务依赖的数据库？
  • CMDB作用
    • 自动化脚本的输入源，部署时，脚本去CMDB查哪些机器待部署，IP是多少
    • 故障影响分析
    • 资产盘点
  • CMDB痛点-数据准确性
    • 人工录入缺乏时效
    • 通过自动发现机制和流程联动
      • 例如，VM克隆之后自动注册到CMDB，保证数据实时准确

举个例子CMDB-自动扩容一个数据库

  • 发布与注册
    • 发布平台-扩容命令
    • 调用CMDB接口，查询剩余资源
    • 新机器初始化之后，发布平台
  • 监控挂载
    • 监控平台轮询CMDB-发现MySQL实例
    • 监控平台自动为MySQL挂载监控模板
  • 故障处理
    • 监控平台告警
    • 告警平台接受信号，查询CMDB-属于哪个业务/谁负责
    • 告警平台执行自动化脚本
      • 自动化平台去CMDB抓取该实例的SSH登录凭证
      • 自动执行SOP

如何在集群中配置CMDB

  • CMDB本身也必须是高可用，可扩展的集群架构
  • 可以基于（蓝鲸配置平台，NetBox）成熟架构部署CMDB
  • 架构设计
    • 存储层
      • 关系型数据库-MySQL/PostgreSQL-存储CIs的元数据，权限，审计日志，利用MGR或主从架构保证数据
      • 图形数据库-Graph DB/Neo4j-处理复杂的拓扑关联查询
    • 接入层
      • 提供RESTful API供jenkins，prometheus等工具调用。通产部署在K8s集群中，通过ingress负载均衡
    • 采集层
      • 运行在物理机/虚拟机上的agent，通过扫描任务定时（snmp，ssh）上报数据
  • 部署步骤
    • 基础设施
      • 用ansible准备好mysql集群和redis缓存
    • 模型定义
      • 类-主机/数据库/中间件
      • 属性-内存/ip/状态/负责人
      • 关联-RunsOn (运行在), BelongsTo (属于), ConnectedTo (连接到)。
    • 数据初始化与自动发现
      • 配置Cloud API接入-自动拉取云主机信息
      • 配置K8s自主发现-实现同步Pod状态
      • 配置脚本采集-通过Shell/Python脚本采集物理机硬件SN码

CMDB跑在Pod中合不合适

“我认为 CMDB 的 API 和前端组件非常适合 Pod 化，因为这能充分利用 K8s 的高可用和水平扩展特性,
但作为 DBA，我会建议将 CMDB 的底层存储（如 MySQL）放在 K8s 集群之外，或者至少放在独立的数据管理集群中。这是为了规避‘循环依赖’风险——确保在业务 K8s 集群发生全站故障（Site-wide Failure）时，我们依然能够访问 CMDB 获取关键的拓扑数据进行容灾恢复。这体现了 SRE 中‘不把鸡蛋放在一个篮子里’的故障域隔离原则。”
