Linux复习_复习框架

发表于2026-04-07|更新于2026-04-07|操作系统Linux面试
|浏览量:

一、 核心概念与“八股文”要点

1. 内核与空间管理

  • 用户态 (User Space) 与内核态 (Kernel Space)： 为什么要划分？（为了安全和稳定）。
  • 系统调用 (System Call)： 用户态如何通过 int 80h 或 syscall 进入内核态。

2. 进程管理（重点）

  • 进程、线程、协程的区别： 资源分配的基本单位 vs CPU 调度的基本单位。
  • 进程状态： R (Running), S (Sleeping), D (Disk Sleep - 不可中断睡眠), Z (Zombie - 僵尸进程), T (Stopped)。
  • 孤儿进程 vs 僵尸进程： 产生原因、危害及清理方法（僵尸进程需找父进程或 kill 父进程）。
  • 调度算法： 完全公平调度器 (CFS) 的基本原理。

3. 内存管理

  • 虚拟内存与物理内存： 页表 (Page Table)、MMU (内存管理单元) 的映射关系。
  • Buffer 与 Cache： Buffer（写缓冲，解决速度不匹配）vs Cache（读缓存，加速读取数据）。
  • SWAP 分区： 触发时机、对性能的影响。
  • OOM Killer： 内核如何根据 oom_score 决定杀掉哪个占用内存过大的进程。

4. 文件系统与 I/O

  • Inode 与 Block： Inode 存储元数据（权限、大小、时间戳），Block 存储实际内容。
  • 硬链接 vs 软链接： Inode 号是否相同、跨文件系统支持情况。
  • VFS (虚拟文件系统)： Linux 如何实现“一切皆文件”。
  • I/O 模型： 阻塞、非阻塞、多路复用（select, poll, epoll 的区别，尤其是 epoll 的红黑树和就绪列表）。

进程监控 ps -ef, pstree, pgrep

三、 高频面试题解析

1. 系统负载 (Load Average) 高，但 CPU 使用率低，可能是什么原因？

  • 分析： Load Average 统计的是处于 R 和 D 状态的进程总数。
  • 原因：
    • I/O 等待： 磁盘读写太慢，大量进程卡在 D 状态（不可中断睡眠）。
    • 锁竞争： 进程在等待某些内核锁。
    • 僵尸进程： 虽然不占 CPU，但在进程表中占位。

2. 发现磁盘空间满了，但 du -sh 发现加起来还没到总容量，为什么？

  • 原因 1： Inode 耗尽。小文件太多，块还有，但 Inode 用光了。
  • 原因 2： 已删除文件未释放句柄。文件被删了（rm），但仍有进程在读写它（句柄未关），空间不会释放。
  • 解决： 使用 lsof | grep deleted 找到相关进程并重启或杀掉。

3. Linux 系统启动流程是怎样的？

  1. BIOS/UEFI 自检。
  2. GRUB 加载内核。
  3. Kernel 初始化硬件、挂载只读根目录。
  4. Systemd (或 Init) 进程启动（PID 1）。
  5. 执行默认 Target（如 multi-user.target）下的服务。

4. 如何查看一个进程打开了哪些文件或网络连接？

  • 命令： lsof -p <PID>。
  • 网络： netstat -p 或 ss -p。

5. 如果服务器响应很慢，你的排查思路是什么？

  1. 看整体： top 查看 CPU、Load、内存。
  2. 看磁盘： iostat 确认是否是磁盘 I/O 瓶颈。
  3. 看网络： sar -n DEV 查看网卡流量，ping 检查延迟。
  4. 看日志： /var/log/messages, dmesg 查看是否有硬件报错或 OOM。
  5. 看进程： 如果是 Java 应用，配合 jstack/jstat 排查。

四、 给你的建议

作为运维工程师，你不仅要会操作，还要有工程化思维：

  • 自动化： 不要只说 chmod，要提你会用 Ansible 批量修改数千台机器的权限。
  • 监控： 不要只说看 top，要提你会配置 Prometheus + Grafana 进行异常预警。
  • 容器化： 现在的 Linux 运维大多与 Docker/K8s 深度绑定，理解 Namespace（隔离环境）和 Cgroups（资源限制）是进阶大厂的必经之路。
