项目简介

《混合云架构的web高可用集群搭建运维》

  • 项⽬地址 ： https://yh1140.github.io/categories/web综合集群
  • 技术栈 ：Rocky 9, K8s, Ansible, Jenkins, MySQL, Redis, NFS, Prometheus, Kafka, ELK, FRP, JumpServer
  • 项⽬描述 ：该项⽬模拟真实⽣产环境，以⾼可⽤，⾃动化，⼀致性，观测性，安全性为⽬的，构建了⼀个混合云架构的服务器集群（云端控制平⾯、防⽕墙节点、k8s业务集群、负载均衡集群、Mysql集群、Redis集群、kafka集群、NAS节点、Prometheus联邦 等 8-15 个节点）。实现了基础环境分钟级交付、流量⾼负载均衡、业务零停机发布、数据强⼀致备份、监控告警、ai运维等功能。为复杂业务提供了可靠的系统底座。
  • 核⼼内容 ：
    • 混合云架构 ：搭建 FRP 隧道 + SSH Proxy + NAT转发 实现云对内主机的安全穿透与集中管理。
    • 安全机制 ：部署jumpserver(安全审计)，Ingress-TLS(传输加密)等提供安全管理。
    • CI/CD ：集成 Zadig + Ansible +GitLab实现代码构建、数据库热迁移及 K8s 业务镜像的声明式更新。
    • IaC：编写 Ansible Playbook，实现节点基础环境与中间件服务的标准化、幂等化交付。
    • 数据可靠 ： 部署 MySQL ⼀致性集群与 NAS 动态存储，实现容器卷按需分配与数据库⾃动化灾备。
    • 流量负载 ： 配置 Keepalived + VIP 消除⽹关单点故障，集成 Ingress-Nginx 实现 L7 流量精准分发。
    • 监控链路 ：构建 Prometheus 联邦+ Grafana 多维可视；Kafka+ELK 中台实现海量⽇志分析告警。
    • 业务韧性 ：基于 K8s 实现多副本编排，联动 HPA +探针 保障业务弹性扩缩容与⾃愈。
    • AIOps：部署 openclaw 驱动 SOP ⾃动排障；时序学习分析预测流量⾼峰提前扩容；关联事件聚合告警；⽇志聚类分析降噪。
  • 项⽬成果：
    • ⾼可⽤：在⾼流量与崩溃模拟中，⼼跳丢失到VIP漂移秒级耗时，Pod30秒扩容，进程崩溃100%重启。
    • 监控：故障发⽣到告警发⽣ < 15秒，ai聚合告警准确率⼤于50%，预测扩容成功率⼤于80%。

项目逻辑架构

  • 云端控制节点

    • 角色：线上运维管理中心；通过内网穿透实现云对内的节点批量管理和部署
    • 组件：Ansible，CI/CD，FRP
    • 指令下发链路：Cloud-Hub (Ansible) -> 防火墙 (DNAT) -> 跳板机/JumpServer -> 各内网节点
    • 优势：云服务器多端连接方便（手机也能连），集中管理，更加规范，减少失误
    • 潜在风险：权限过大，如果SSH私钥泄露，整个内网集群直接裸奔

  • 边缘防御层

    • 角色：内网环境护城河；外部流量进入私网的第一道关卡，内部服务器访问互联网的唯一出口
    • 组件：两台防火墙节点Firewall Nodes（VIP+keepalived），iptables，NAT，ACL，WAN/LAN
    • 安全设计：
      • 连接跟踪表优化：记录每一个通过的连接，优化内核参数net.netfilter.nf_conntrack_max预防高并发下灌满
      • 管理端口隐藏：可以不开放22端口，修改为高位端口（2222），只允许云节点的ip访问
      • 反向路径过滤：开启内核的反向路径过滤，防止 IP 欺骗攻击（Spoofing），确保进入防火墙的包确实是从它声称的接口进来的
    • 流量路径：
      • 请求进入：用户访问 www.yyyh.com -> 解析到公网 VIP -> 到达 FW-01
      • 初步清洗：防火墙检查包，发现是 80 端口，属于放行名单；发现是恶意攻击 IP，直接丢弃
      • 地址转换：防火墙通过 DNAT，把请求包的目标地址从“公网 VIP”改为“内网 LB VIP”
      • 内网转发：包进入内网，交给 HAProxy
      • 相应回传：后端处理完后，回包给防火墙，防火墙再通过反向转换发给用户

  • 安全管理层

    • 角色：跳板机，内网节点唯一访问入口，部署jumpserver实现操作审计和资产管理
    • 组件：DMZ，jumpserver，
    • 功能实现：身份校验，权限切分，指令拦截与录像

  • 负载均衡层

    • 角色：流量调度中心；四层/七层负载均衡，将数以万计的用户请求，按照预设的策略，均匀、精准地分发给后端的业务服务器
    • 组件：四层（HAProxy/LVS），七层（Ingress-Nginx）
    • 优势：
      • 性能分层：四层负载基于内核处理，新能强于七层，作为整个集群的流量缓冲
      • SSL集中管理：在四层LB作灰度发布或 DDoS 限流，而让 Ingress 专注于 Kubernetes 内部的服务发现
      • 故障隔离：

  • 业务容器层

    • 角色：业务中心，保证业务高可用，高效利用资源，节约成本
    • 组件：k8s集群（探针，HPA，等K8s核心机制）
    • 功能：
      • 应用托管：将web应用打包为容器镜像并运行
      • 弹性伸缩：根据流量大小，秒级增加或减少Pod数量
      • 故障自愈：如果某个容器挂了，K8s在健康节点上面新起一个
      • 服务发现：暴露容器IP给负载均衡层
      • 滚动更新：

  • 数据层

    • 角色：持久化数据存储
    • 组件：MySQL MGR集群（GTID），Redis 高速缓存集群（哨兵或3主3从），Kafka 消息缓冲集群（kraft），NAS 节点，Harbor 镜像仓库
    • 功能：
      • 数据库高可用：GTID+MGR 实现数据强一致和事务断点续传
      • 存储共享：NFS实现web静态一致，动态供给Pod实现解耦
      • 备份：定时推送数据库备份文件到NAS存储，
    • 优化：
      • 引入 Longhorn 或 Ceph 等分布式存储，数据切片存储在多worker节点，消除NAS单点故障

  • 监控与日志分析层

    • 角色：实现集群的全链路监控和日志分析
    • 组件：Node_exporter，K8s监控组件(kube-state-metrics，cadvisor)，Prometheus联邦，Grafana，Altermanager，日志采集分析（filebeat，ELK）

数据流向路径

  • Redis缓存链路

    • 路径：Web Pod <–> Redis Cluster/Sentinel
    • 机制：Cache-Aside（旁路缓存）策略。
        • 读取：Web 程序先去 Redis 查数据（如 Session）。如果命中了，直接返回，毫秒级响应。
      • 写入：数据变更时，先更新 MySQL，再删除或更新 Redis。
    • 意义：通过内存操作大幅提升并发能力，保护后端数据库不被瞬间请求冲垮。

  • MySQL MGR 链路

    • 路径：Web Pod $\rightarrow$ MySQL VIP/Proxy $\rightarrow$ MGR Primary Node $\rightarrow$ MGR Secondary Nodes
    • 机制：Paxos 一致性协议 + GTID
      • 写入流程：当 Web Pod 执行一条 INSERT 时，MGR 主节点会将该事务发送给所有节点。
      • 认证机制：只要集群中超过半数节点（如 3 节点中的 2 个）确认收到了日志并同意执行，该事务才会提交。
      • GTID（全局事务 ID）：确保每个事务在整个集群中都有唯一身份证，防止主从数据不一致。

  • 日志分析链路

    • 路径：Web Pod (Filebeat) $\rightarrow$ Kafka $\rightarrow$ Logstash $\rightarrow$ Elasticsearch $\rightarrow$ Kibana
    • 机制：
      • Filebeat (Sidecar 模式)：作为 Web Pod 的“小跟班”，实时监控 /var/log/nginx/access.log 的变化。
      • Kafka (生产者模式)：Filebeat 把日志“丢”进 Kafka 的 Topic 里。这里起到了削峰填谷的作用——即便 Web 每秒产生 10 万条日志，Kafka 也能先存下来。
      • Logstash (消费者模式)：Logstash 从 Kafka 订阅消息。它的任务是“清洗”：把杂乱的字符串拆解成 timestamp, ip, status, request_time 等字段。
      • Elasticsearch (存储与索引)：清洗后的结构化数据存入 ES。ES 会对数据进行分词和索引，实现近实时的海量搜索。
      • Kibana (展示)：你通过 Kibana 的图形界面，一键生成“24 小时访问量趋势图”。

  • 存储共享链路

    • 路径：Web Pod $\longleftrightarrow$ NFS-Provisioner $\longleftrightarrow$ NAS 服务器 ($192.168.31.7$)
    • 核心机制：
      • RPC (远程过程调用)：Linux 内核通过 NFS 协议，将远程 NAS 上的文件夹虚拟化为本地目录。
      • K8s 动态供给 (StorageClass)：你不需要手动去 NAS 建文件夹。当你提交一个 PVC 申请时，NFS-Provisioner 会自动在 NAS 上创建一个以 pvc-xxxx 命名的目录，并动态挂载给 Pod。
      • 备份链路：MySQL 的定时任务脚本通过 nfs-client 挂载 NAS 的 /web/html/db_bak 目录。备份完成后，.sql.gz 文件直接写入 NAS。
