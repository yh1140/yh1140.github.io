K8s概念_集群架构与组件

发表于2026-04-17|更新于2026-04-17| k8s
|浏览量:

集群架构

一个完整K8s集群会包括

  • 控制平面
  • 工作节点
  • 额外插件
  • 三大标准接口
  • 基础设施支撑
    • 外部负载均衡器
    • 镜像仓库
    • 外部存储后端
  • 扩展与自定义逻辑
    • CRD
    • Operator
  • 安全与治理组件
    • 密钥管理
    • 准入策略引擎
  • 客户端与生态工具
    • Helm
    • Kustomize

集群架构的设计不是一成不变的，而是根据可靠性需求、资源预算和业务场景演变出多种形态

按控制平面数量划分

  • 单主架构 Single Master

    • 1个Master节点+多个Worker节点
    • 部署简单，资源消耗少
    • 缺点：单点故障

  • 多主高可用架构 Multi-Master HA

    • 结构：3-5奇数个的Master节点
    • 特点：利用负载均衡（LB）和选主机制

按etcd存储位置划分

在高可用架构中，etcd（数据库）放哪里是区分“专业程度”的关键

  • 堆叠etcd模式-Stacked etcd

    • 设计：etcd运行在每一个Master节点上，与API Server强绑定
    • 优点：通过kubeadm一件创建，管理简单
    • 缺点：API Server 与 etcd 争夺资源。如果Master 压力大会影响数据库一致性

  • 外部etcd模式-External etcd

    • 设计：将etcd独立出来，组成一个专门的数据库集群。API Server远程连接这个集群
    • 优点：解耦
    • 缺点：资源消耗

按部署环境划分

  • 云原生托管架构（Managed K8s, 如 ACK/EKS/GKE）

    • 直接出钱购买
    • 分工：云厂商负责控制平面的高可用、升级和维护，你完全看不到 Master 节点，只能看到 Worker 节点。
    • 意义：运维负担降到最低，支持自动弹性伸缩（Cluster Autoscaler）。

  • 裸机/私有云架构（On-Premise / Bare Metal）

    • 核心逻辑：所有组件（从底层系统到网络插件）全部自建。
    • 挑战：需要自己解决 Load Balancer（如 Keepalived+LVS）、存储（Ceph/NFS）和网络协议（BGP/隧道）。

  • 边缘计算/轻量化架构（Edge K8s, 如 K3s）

    • 做法：将所有组件打包成一个极小的二进制文件，甚至支持使用 SQLite 替换 etcd。
    • 场景：树莓派、工控机、IoT 设备等内存极小的环境

复杂环境

  • 跨可用区架构（Multi-Zone）：Master 节点分布在同一个城市的不同机房，防止整个机房断电。

  • 混合云架构（Hybrid Cloud）：

    • 公有云管理私有云：控制平面在云端（利用其高带宽和稳定性），Worker 节点在本地机房（保护数据隐私或利用闲置硬件）。
    • 思路：通过 VPN 或专线打通网络，实现资源的跨云调度。

  • 多集群架构（Multi-Cluster / Federation）：不是一个大集群，而是多个独立集群通过 Karmada 或 OCM 等工具进行统一分发

控制平面组件

  • kube-apiserver

    • REST接口：提供 HTTP API，无论是 kubectl、Dashboard 还是集群内部组件，都通过它通信。
    • 安全把关：负责身份认证（你是谁）、授权（你能干什么）和准入控制。
    • 中转站：它是唯一能直接读写 etcd 的组件
    • 无状态：可以启动多个实例，通过负载均衡实现高可用

  • etcd

    • 高可用的键值数据库
    • 状态存储：保存了集群所有的配置数据和运行状态（比如：现在开了几个 Pod、它们在哪个 IP）
    • 一致性保证：基于 Raft 协议，确保在多节点部署时数据绝对同步
    • 伸缩特性：分布式集群。通常部署 3/5 个节点，必须保证奇数以防止脑裂

  • kube-scheduler

    • 资源调度，负责为新创建的 Pod 选择合适的节点
    • 过滤（Filtering）：排除掉资源不足（CPU/内存不够）或不满足约束（亲和性、污点）的节点
    • 打分（Scoring）：在剩下的节点中，选出一个最“优”的节点

  • kube-controller-manager

    • 各种控制器的集合，负责维持集群的“期望状态”
    • 节点控制器：监控 Node 是否掉线
    • 副本控制器：如果你要求运行 3 个副本，挂了一个，它就负责再起一个
    • 端点（Endpoints）控制器：维护 Service 和 Pod 的对应关系
    • 伸缩特性：主备模式。同样通过选主机制保证只有一个实例在下发指令

  • cloud-controller-manager

    • 将 K8s 与具体的云厂商（阿里云、AWS 等）API 挂钩
    • 负载均衡管理：自动创建云端的 SLB/ELB
    • 路由管理：配置云网络里的路由表

工作逻辑-控制循环

控制平面组件之间的工作逻辑在于声明式而不是命令式

  1. 用户：通过 apiserver 向 etcd 存入一个愿望（比如：我要跑 3 个 Nginx）
  2. Controller Manager：发现 etcd 里的愿望是 3 个，但实际只有 0 个，于是请求 apiserver 创建 Pod
  3. Scheduler：发现有新的 Pod 还没分配机器，计算后决定把它们塞到 Node-01
  4. Kubelet (Node端)：看到 apiserver 分配给自己任务了，开始拉镜像、跑容器

控制平面部署选项

控制平面可以通过以下几种方式部署

  • 传统部署

    • 控制平面组件直接在专用机器或虚拟机上运行，通常作为 systemd 服务进行管理

  • 静态Pod

    • 控制平面组件作为静态 Pod 部署，由特定节点上的 kubelet 管理。 这是像 kubeadm 这样的工具常用的方法

  • 自托管

    • 控制平面在 Kubernetes 集群本身内部作为 Pod 运行， 由 Deployments、StatefulSets 或其他 Kubernetes 原语管理。

  • 托管K8s服务

    • 云平台通常将控制平面抽象出来，将其组件作为其服务的一部分进行管理

工作节点组件

  • kubelet

    • 每个节点的代理程序
    • 管理Pod里面的容器
    • 状态汇报：它会不断向 API Server 汇报自己的健康状况、资源占用（CPU/内存）以及 Pod 的运行状态
    • Pod 生命周期管理：当 API Server 说“在这个节点跑个 Pod”时，kubelet 负责调用容器运行时去拉镜像并启动
    • 健康检查：它会根据我们在 YAML 里定义的 Liveness/Readiness Probe 来盯着容器。如果容器死掉了，它负责重启

  • kube-proxy

    • 维护节点的网络规则
    • 不直接传输数据，而是配置底层路径
    • 服务发现与负载均衡：它是 Service 机制的实现者。当一个请求访问 Service IP 时，由 kube-proxy 决定将流量转发到后端哪一个具体的 Pod 上
    • 网络规则维护：它会在宿主机上操作 iptables 或 ipvs，确保跨节点、跨 Pod 的通信能够跑通

  • 容器运行时

    • 负责运行容器
    • 执行动作：真正的拉取镜像、解压镜像、创建并启动容器进程
    • K8s通过CRI协议与他对接，使得K8s可以支持不同的容器技术

工作逻辑

组件之间接力启动一个Pod

  1. 控制平面：Scheduler 选好了 Node，API Server 把指令发给该 Node 的 kubelet
  2. kubelet：收到指令，转头跟 Container Runtime 说：“嘿，帮我按这个图纸（Pod Spec）起几个容器。”
  3. Container Runtime：去仓库（镜像仓库）把镜像拉下来，启动容器，并把结果告诉 kubelet
  4. CNI 插件：在容器启动时，负责给它插上“网线”，分好 IP
  5. kube-proxy：更新本地的“路牌”（iptables/ipvs），让别人知道新来的这个 Pod 怎么访问
  6. kubelet：回头告诉 API Server：“报告！任务已完成，Pod 正在运行。”

插件

  • 网络插件-CNI

    • 连通性：确保集群内所有的 Pod 无论在哪个节点，都能通过 IP 直接互相访问
    • IP 管理：为每个新创建的 Pod 分配一个唯一的集群内 IP
    • 常见实现
      • Flannel：最简单，适合学习和小型测试
      • Calico：功能强大，支持网络策略（防火墙），性能优异，生产主流
      • Cilium：基于 eBPF，性能天花板，具备高级安全和监控功能

  • DNS插件-CoreDNS

    • 服务发现：你不需要背 Pod 的 IP（因为 IP 经常变），只需要访问 Service 的名称（如 my-db），CoreDNS 会自动帮你解析到对应的 IP 上
    • 从 K8s 1.11 开始，CoreDNS 是官方默认且必须安装的插件

  • 服务暴露插件-Ingress Controller

    • 统一入口：Service 只能暴露简单的 IP/端口。Ingress 允许你通过域名（如 api.example.com）和 路径（如 /v1）来转发流量
    • 负载均衡：在应用层（第七层）进行流量分发，支持 SSL 证书卸载
    • 常见实现
      • Nginx Ingress Controller (最常用)
      • Traefik
      • Istio Gateway

  • 资源监控插件-Metrics Server

    • 数据采集：收集节点和 Pod 的实时 CPU、内存利用率
    • 驱动功能：它是 kubectl top 命令的数据源，也是 HPA（自动横向扩容） 工作的依据。如果没有它，K8s 就不知道什么时候该增加副本

  • 可视化界面-Dashboard

    • 图形化管理：通过网页浏览器查看集群状态、部署应用、查看日志，不需要一直对着黑色的命令行终端

  • 日志收集-Promtail / Fluentd

    • 搬运容器日志

三大接口

CRI (Container Runtime Interface) - 容器运行时接口

  • 职责：负责“生孩子”。K8s 告诉它“我要跑个 Nginx”，CRI 就去拉镜像、起容器。
  • 地位：最底层。没有它，Pod 只是 YAML 文件里的文字，无法变成运行的进程。
  • 代表实现：containerd (你目前 1.35 用的)、CRI-O。

CSI (Container Storage Interface) - 容器存储接口

  • 职责：负责“找硬盘”。Pod 漂移到哪，CSI 就负责把云硬盘或 NAS 挂载到哪。
  • 地位：数据的命脉。它解耦了 K8s 与各种云厂商（阿里云、AWS）的存储协议。
  • 代表实现：Ceph CSI、NFS CSI、各云厂商磁盘插件。

CNI (Container Network Interface) - 容器网络接口

  • 职责：负责“拉网线”。给 Pod 发 IP 地址，打通 Pod 之间的隧道。
  • 地位：通信基础。没有它，Pod 之间就是孤岛，Service 和 Ingress 也无法工作。
  • 代表实现：Flannel、Calico、Cilium。

K8s-API

  • 在 K8s 中，“一切皆对象，一切皆 API”。
  • K8s API 的本质不是为了“执行命令”，而是为了 维护状态
  • K8s api 是唯一入口： 整个集群中只有 API Server 能读写 etcd。
  • K8s api 是标准化的契约：无论底层是 Ubuntu 还是 CentOS，无论网络是 Flannel 还是 Cilium，对外的 API 永远长得一样。
  • K8s api 是可扩展的：通过 CRD（自定义资源），你可以像定义原生的 Pod 一样定义你自己的“数据库对象”或“AI任务对象”

API分层过滤

每当你发起一个 API 请求（无论是通过 kubectl 还是代码），都会经过 API Server 的一套标准流水线：

  1. 认证 (Authentication)：你是谁？（检查证书、Token）。
  2. 鉴权 (Authorization)：你能干什么？（RBAC 检查，是否有权操作这个 Namespace）。
  3. 准入控制 (Admission Control)：你发的东西合规吗？（例如：检查 Pod 是否超过了资源配额，或者自动给 Pod 加上标签）。
  4. 持久化：通过校验后，数据最终存入 etcd

API逻辑结构

为了处理成百上千种资源，K8s 将 API 组织成了树状结构：

  1. 资源路径：
    /apis/apps/v1/namespaces/default/deployments/my-nginx
    • apis: 告诉 Server 这是个命名组 API。
    • apps: API 组 (Group)。按功能分类（如存储、网络、应用）。
    • v1: 版本 (Version)。区分 Alpha, Beta, Stable。
    • deployments: 资源类型 (Resource)。
    • my-nginx: 资源名称 (Name)。
  2. 发现机制：
    • API Discovery: 就像餐厅的“菜单”。它告诉客户端这个集群支持哪些版本的 API。
    • OpenAPI (v2/v3): 就像菜单上的“配料表”。它详细定义了每个字段（如 replicas 必须是整数）的校验规则。
