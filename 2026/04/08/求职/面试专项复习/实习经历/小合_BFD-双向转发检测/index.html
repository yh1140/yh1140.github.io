Bidirectional Forwarding Detection

BFD解决的是调度系统中最致命的问题：链路感知的滞后性

简介

BFD 是一种全网统一的、检测链路故障的轻量级协议。它的核心任务非常简单：以毫秒级的速度探测两个网络节点之间的路径是否通畅。

如果某条 100G 的出口链路物理上没断，但发生了“灰度故障”（如光衰导致严重掉包），SNMP 轮询可能要几分钟才能发现，而 BFD 在 几百毫秒 内就能触发调度切换。

工作原理

BFD 的逻辑非常严谨，类似于两人对话确认生存：

  1. 建立会话： A 和 B 协商发送心跳包的时间间隔（例如 100ms）。
  2. 持续握手： A 发给 B，B 发给 A。只要在约定的时间内（如 3 次间隔内）收到对方的包，就认为链路正常。
  3. 快速崩溃： 一旦超时未收到包，BFD 状态立即变为 Down，并瞬间通知上层协议（静态路由、BGP、OSPF 或你的自动化调度脚本）。

故障排查逻辑

如果你在调度监控中看到 BFD 相关的报错，请遵循以下严谨的排查思路：

报错信息： BFD Session Down: Neighbor 10.1.1.2 unreachable 直译内容： BFD 会话关闭：邻居 10.1.1.2 不可达。 解决思路：

  1. 链路检查： 检查底层物理链路是否真的发生中断或高延迟。
  2. 配置一致性： BFD 要求两端协商。检查本端和对端（运营商端）的 discriminator（标识符）是否互为镜像（即我的 local 是你的 remote）。
  3. CPU 保护： 大带宽下（如你的 1T 节点），如果设备 CPU 过高，可能导致 BFD 心跳包处理延迟，产生误报。此时应调大 multiplier（容错倍数）或时间间隔。
  4. ACL/防火墙： 确认没有误封 BFD 协议包（通常使用 UDP 端口 3784）。

BFD 在调度中的三种核心角色

A. BFD + 静态路由 (最简单的“主备切换”)

在出口数量较少时，调度器通常配置静态路由。

  • 痛点： 静态路由无法感知远端链路故障（下一跳没挂，但再往后挂了）。
  • 调度表现： 将静态路由关联 BFD。一旦 BFD 检测到异常，静态路由立刻失效，流量自动切往优先级较低的备用出口。

B. BFD + BGP (ISP/CDN 调度主流)

在你提到的 1T 级别节点中，BGP 是核心。

  • 痛点： BGP 的 Keepalive 机制默认是 60s，确认失效要 180s。这对 95 计费的带宽点数管理来说太慢了。
  • 调度表现： BGP 挂载 BFD 后，检测时间从 分钟级 缩短到 毫秒级。这能防止因为单链路故障导致的“流量局部打爆”和“无效打峰点数消耗”。

C. BFD + PBR (策略路由)

  • 场景： 针对特定业务 IP 的调度。
  • 调度表现： 当调度系统通过 PBR 将流量引向某个口时，如果该口链路抖动，BFD 联动 PBR 能够实现“毫秒级避障”。
