基础设施与连接

①FRP 隧道的基础工作原理

  • 问题： 你的项目涉及到公有云（火山云）和私有主机的混合架构。请简单描述一下 FRP 隧道的基础工作原理。在你的配置中，FRP 的 Server 端和 Client 端分别部署在哪里？如果我想通过公有云的 80 端口访问私有云里的 Web 服务，在 FRP 的配置文件里最核心的几个参数是什么？
  • 回答： FRP 的工作原理是利用反向代理技术，在拥有公网 IP 的服务端（frps）与内网客户端（frpc）之间建立一条加密的控制隧道，从而实现内网服务的公网映射。在本项目中，Server 端部署在火山云公有云节点作为流量接入枢纽，Client 端部署在私有 Rocky Linux 宿主机；若要实现公网 80 端口访问内网 Web，核心配置参数包括 [common] 段的 server_addr（服务端公网 IP）、server_port（隧道通信端口）和 token（安全校验密钥），以及业务模块中的 type = http（协议类型）、local_port = 80（内网服务实际端口）和 custom_domains（绑定的访问域名或公网 IP）。

②FRP自动恢复&高可用

  • 问题：如果 frpc 所在的主机重启了，你怎么保证隧道自动恢复
  • 回答：在基础实现上，我利用 Systemd 对 frpc 进行了服务化管理，配置了 Restart=on-failure。同时，FRP 客户端自带重连机制（login_fail_exit = false），确保网络波动后能自动找回服务端
  • 问：如何保障frp隧道的高可用
  • 答：
    • 使用frp是出于成本和环境， 在真正生产环境可以使用云厂商的私网链接，或者vpn专线。
    • frp加密传输+身份验证

③SNAT/DNAT

  • 问题： 在你的混合云架构中，私有云里的主机本身是没有公网 IP 的。如果这些主机需要主动访问互联网（比如执行 dnf update 或下载 Docker 镜像），你是如何通过防火墙节点实现 SNAT 的？请简述 iptables 或 nftables 实现这一过程的关键步骤。
  • 回答：“在混合云架构中，实现私有主机主动出网的核心是利用防火墙节点的 SNAT（源地址转换） 功能。具体实现上，首先需开启 Linux 内核的 ip_forward 转发权限，然后在 iptables 的 nat 表 POSTROUTING 链中配置 MASQUERADE 或 SNAT 规则；当内网报文经过防火墙外网卡流出时，内核会将其私网源 IP 替换为防火墙的公网 IP，并利用 连接追踪（conntrack） 机制维护映射表，从而确保互联网回程报文能准确送达内网主机。”

④tcpdump/Wireshark排障

  • 问题： 假设现在用户反馈访问 Web 服务非常慢，甚至偶尔出现 504 超时。你会如何在 火山云的 FRP 服务端 和 私有云的 FRP 客户端 两个位置使用 tcpdump 进行定位？你重点会观察 TCP 三次握手中的哪些标志位（Flags）或时间戳？
  • 回答：“针对 504 超时，我会采用双端同步抓包法：在火山云服务端抓取 80 端口和隧道端口流量，在私有云客户端抓取隧道和本地服务端口流量。通过对比同一 Sequence Number 数据包在两端出现的绝对时间戳差值来定位延迟段；重点观察 [SYN] 重传判断链路连通性，监控 [RST] 标志确认连接重置原因，并计算 Request 到 Response 的 Delta Time 以区分是‘跨云隧道网络拥塞’还是‘后端业务处理过慢’导致的超时。”

⑤TCP Keepalive

  • 问题： FRP 建立的是长连接。在复杂的跨地域网络中，运营商可能会中途断开长时间无流量的 TCP 连接。你在 FRP 配置中是如何处理“死连接”的？有没有用到 心跳机制（Heartbeat）？
  • 回答：“为了防止运营商由于超时回收‘静默’的 TCP 连接，我在 FRP 配置中启用了应用层心跳机制，通过设置 heartbeat_interval 为 30 秒来保持链路活跃，并配合 heartbeat_timeout 确保在链路失效时 frpc 能立即感知并触发自动重连；此外，我还开启了 TCP 多路复用 (tcp_mux) 减少连接开销，在极端网络环境下会考虑切换 KCP/QUIC 协议，利用 UDP 的特性来绕过中间网络节点的拦截，从而保证混合云链路的持久稳定性。”

自动化交付

①幂等性

  • 问题： 你提到实现了“分钟级幂等交付”。请从基础实现角度解释一下什么是 “幂等性（Idempotency）”？在编写 Ansible Playbook 时，你是如何确保如果你连续运行两次脚本，第二次不会因为资源已存在而报错或者产生重复配置的？请举一个具体的模块例子（如 file 或 yum/dnf）。
  • 回答： “幂等性是指多次执行同一自动化任务，系统最终状态保持不变且无副作用。在 Ansible 中，它是通过声明式管理实现的：模块在执行前会先通过 Check 阶段对比资源的‘实际状态’与‘期望状态’，仅在二者不一致时才触发变更。例如使用 dnf 模块配置 state: present 时，若软件包已安装，Ansible 会直接返回 OK 而非重复执行安装指令，从而确保了脚本在连续运行下的安全性和稳定性。”

②机密数据安全

  • 问题： 在 Playbook 中，你是直接明文编写这些密码吗？如果是存放在 GitHub 或公司 GitLab 上，如何防止密码泄露？你是否了解或使用过 Ansible Vault 对变量进行加密？
  • 回答： “在自动化交付中，严禁明文编写机密数据。我采用了 Ansible Vault 机制，通过 AES-256 对称加密将 MySQL 密码、FRP Token 等敏感变量统一存放在独立的 secrets.yml 中并进行文件加密。在 Git 提交流程上，我利用 变量分离 确保仓库中仅存在密文，并配合 .gitignore 严格禁止存放解密密钥的 .vault_pass 文件上传。执行时通过 --vault-password-file 实现无人值守的安全解密，从而确保了机密信息在代码全生命周期内的安全性。”

③复杂逻辑与错误处理

  • 问题： 如果在执行某一步（如安装 K3s）时因为网络波动失败了，Ansible 默认会停止后续任务。你如何确保系统能够执行清理工作？有没有用过 block, rescue 和 always 语法？或者如何利用 failed_when 来定义什么才算真正的失败？
  • 回答： “为了确保自动化流程的健壮性，我采用了 Block/Rescue/Always 语法进行异常分层管理：将安装核心逻辑放入 block，在 rescue 中编写清理残留文件和回滚配置的逻辑，确保失败后环境的干净，并通过 always 处理临时文件的回收。同时，针对复杂任务，我会利用 failed_when 结合正则表达式捕获命令输出（stdout）中的关键字，从而实现比单纯依赖返回码更精准的状态监控。对于网络波动，我还会配合 retries 机制进行自动重试，以最大程度减少偶发性网络错误对分钟级交付的影响。”

核心存储高可用

①MGR 架构

  • 问题： 你选择了 MySQL 8.4 版本的 MGR 架构。请问 MGR 在基础实现上，它是靠什么机制来保证多个节点之间的数据强一致性的？（提示：可以提到 Paxos 协议或组通信机制）。另外，在 MGR 集群中，如果其中一个节点宕机了，外部的 Web 应用是如何感知并实现自动切换连接的？
  • 回答： “MGR 的数据强一致性是基于 Paxos 分布式共识协议 实现的，通过 GCS 组通信系统 确保所有节点对事务执行顺序达成全局一致，并通过冲突认证（Certification）机制解决并发冲突。在 Web 应用接入层面，我通常配合 MySQL Router 或 ProxySQL 等中间件实现故障感知；当主节点宕机引发集群重新选主时，中间件会自动探测到成员状态变更并将流量切换至新主节点，从而实现对应用层透明的自动故障转移。”

流量接入与分发

①

  • 问题： 在你的 Web 集群中，外部请求是如何进入到容器内部的？请描述一下从 Ingress 控制器到 Service，再到具体 Pod 的流量转发路径。在这个过程中，Ingress 和 Service 分别起到了什么作用？
  • 回答： 在我的集群架构中，外部请求首先通过混合云环境下的 FRP 隧道或本地解析到达 LB 负载均衡节点。由于 Ingress-Nginx 控制器采用了 hostNetwork 模式直接绑定了物理机的 80/443 端口，它能第一时间拦截流量，并根据预设的 Ingress 路由规则进行域名（如 www.yyyh.com）和路径的逻辑匹配。一旦匹配成功，流量会被转发至集群内部对应的 Service，并最终由后端的 Nginx 业务 Pod 进行处理，这些 Pod 通过挂载 NFS 共享存储实现了网页数据的高统一性与持久化。
