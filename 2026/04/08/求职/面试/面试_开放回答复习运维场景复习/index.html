①

运维面试必问：网站突然变慢你要怎么排查
“网站突然变慢”
可能很多人一上来就说一些排查的命令：
top/netstat
虽然没错，但一下子就能被面试官看出来你没有实际经验。

🧨正确的方式：必须有排查思路
我会先判断：慢在哪里

① 是所有用户慢？
② 还是只有某个接口慢？
③ 什么时候开始的，是刚刚变慢，还是一直慢？

👉 👉 必须先找到最可能的故障方向

接下来再这样排查：按调用链路去层层分析
用户 → CDN → 负载 → 应用 → 数据库

👉 👉重点是要一层一层地排查

🧨实战思路
情况1：全部都慢
我第一反应：服务器扛不住了
这时候直接看 CPU / 内存
（很多时候就是 CPU 100%）

情况2：某个接口慢
直接怀疑数据库
有没有慢查询
SQL 有没有走索引

情况3：只有部分地区慢
基本就是网络/CDN问题

情况4：突然变慢
必问两件事：
刚刚有没有上线？
流量是不是暴涨了？

🧨面试重点
不是你会多少命令
👉 而是你有没有排查思路：
✔ 会分情况
✔ 会缩小范围
✔ 有经验进行判断

下面再分享一些故障例子：
一个 SQL 没有走索引 → 数据库拉爆
CPU 被打满 → 服务崩溃
第三方接口慢 → 整个接口超时

②

运维面试： 重启后问题消失说明什么？
“你遇到过重启就好了的情况吗？说明什么？”
可能很多人这时的回答就捉不到重点了

这个问题最关键的在于：
重启不是解决问题，而是掩盖问题

如果在面试的时候能答上这个，就代表你有正确的排查问题思路了。

🧨那到底说明什么？
👉 一般说明：
系统里有状态型问题
例如
✔ 资源被占满
✔ 某些连接/内存没释放
✔ 程序进入异常状态
👉 重启就等于全部清空 ，所以看起来就好了

🧨常见真实原因
原因一：内存泄漏
程序一直吃内存 → 最后卡死
重启后内存清空 → 恢复正常

原因二：连接没释放
数据库连接 / socket 一直占着
连接池被耗尽 → 服务不可用

原因三：线程/死锁问题
线程卡死、互相等待
重启后线程全部重建

原因四：文件句柄耗尽（加分点）
打开太多文件没关闭
新请求直接失败
🧨 面试加分关键
重启只是临时恢复，不是解决问题
真正要做的是
① 找日志
② 看监控
③ 复现问题
④ 定位根本原因
因为不彻底解决问题的话，问题只会在下一次再次发现。
但在实际工作中，有时为了用户体验不得不采取临时恢复的措施，排查根因的工作需要在事后进行，但注意保存好环境。

这样一段说出来，就很能展现你的经验了。

🧨实际例子
最后再分享一些故障例子：
Java 内存泄漏 → 每天都要重启一次
数据库连接没释放 → 服务卡死
线程池满了 → 请求阻塞

③

运维面试：日志没报错 但系统就是异常
“系统响应异常，但查询日志完全没有报错情况，这时你怎么排查？”
很多人这时第一反应就是，这怎么查？要不重启试试？

其实，面试官就是想通过这道题，考察你是否具备全局排查思维和底层系统功底。

💡 核心排查思路：跳出只看日志的思维定势！
1️⃣ 第一步：怀疑日志本身出了问题
你可能没留意到的是，系统异常时日志可能根本没写进去：
磁盘满了/Inode耗尽
日志级别被改了
写日志的线程卡死了
这是最容易出现又是最低级的错误，但也是最能体现你经验的一步。

2️⃣ 第二步：排查底层系统资源瓶颈
如果应用没报错，那就要考虑操作系统层面了
网络连接数打满：查一下TCP状态（netstat -anp）看看是否端口耗尽
文件句柄超出限制
内存 OOM 前夕
磁盘I/O瓶颈：iostat 看看IO是否很高，都在等读写。

一般只有网络连接数和IO瓶颈出现的频率会大一点。

3️⃣ 第三步：排查应用内部状态
这一步就看代码逻辑有没有卡住：
死锁：用jstack（Java）或者 pprof（Go）抓取线程快照看状态。
连接池耗尽：排队等待获取连接，但还没到超时时间所以没报错。
频繁的垃圾回收（GC）：比如Java应用发生频繁的 Full GC，整个程序处于暂停状态，当然不会打日志。

4️⃣ 第四步：排查外部依赖与链路
你的服务没问题，不代表整个调用链没问题：
网络拦截/防火墙：排查安全组、iptables规则是否被改了？包根本没走到你的应用。
DNS解析问题：调用的第三方API域名解析失败或变慢。
中间件阻塞：比如消息队列消费阻塞，或者下游的MySQL发生了慢查询锁表。

🧨最后一个加分关键

回答完上面的排查点后，最后补上这一句，会更能体现你的实战经验：

为了避免以后再出现这种被动局面，我会建议团队完善 APM全链路追踪（如SkyWalking），让请求的瓶颈一目了然，大大加快排查的进度。

③

AI运维提效 海量日志如何破解token爆炸
🕳️ AI+日志分析的难点：
很多运维应该都想过用AI去分析日志吧？
但往往面对的第一个问题就是：
“每天的日志都是海量的几十万行？这分析一下token不直接爆炸了？”
“而且llm响应的时间也很长，分析速度哪里跟得上。”

所以解决问题的核心就是：
怎么把日志压缩成模型能接受的程度？

🧨 一个简单的方法
真实的日志往往无用重复的信息很多
所以我的做法其实很直接：
只把异常相关的日志提出来。

🧨 一、先判断“是不是异常”

这一步只是粗筛。
① 先看日志级别
如果是：
ERROR/FATAL
那基本可以直接认为是异常。
② 再看日志内容
匹配这些关键词：
exception
panic
timeout
connection refused
out of memory
failed / error
只要命中，就进入下一步分析。
这一阶段的目标只有一个：
把可能异常的日志筛出来。

🧨 二、再做异常分类

分类要按 特征强弱排序。
推荐顺序：
OOM→Timeout→ConnectionFailure→
Exception→Fatal / Error→Unknown

例如：
connection refused
即使日志级别是 ERROR，也更适合归类为：
ConnectionFailure
而不是简单的 Error。

🧨 三、合并同一次异常
具体规则是：
只要满足以下情况之一，就认为该行仍然是当前异常的一部分：
该行命中了 stack_trace_pattern，也就是典型的堆栈信息，例如at …、File …, line … 等。
该行属于异常的扩展说明，例如Caused by:、Traceback、root cause 等，这些通常是在解释异常原因。
该行没有表现出新的异常起点特征，也就是说它不像一条新的独立异常日志。
合并后就能直接发给大模型进行分析了。

📌 总结一下整个流程
1、规则拆分
2、异常初筛
3、统一分类
4、合并同一次异常
5、发给大模型

当日志足够干净时，就既能让大模型参与分析，又节省token了。
