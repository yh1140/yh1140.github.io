2.对编译型语言和非编译型语言有了解吗？这两类有哪些语言？
- 编译型语言（如 C、Go）在运行前需通过编译器一次性转换为机器码，执行效率极高但跨平台稍显复杂，常用于 K8s 等对性能要求苛刻的底层架构；而非编译型即解释型语言（如 Python、Shell）在运行时由解释器逐行翻译，开发效率高且部署灵活，非常适合我日常处理的大数据分析和自动化运维脚本。这种底层差异决定了我在实际项目中，会根据业务对“运行性能”与“迭代速度”的不同侧重来选择最合适的工具。
3.虚拟机部署和容器化部署的区别
- 这两者的本质区别在于抽象层级的不同：
1. 虚拟机 (VM)：它是硬件级虚拟化。每个虚拟机都包含完整的操作系统（Guest OS），因此隔离性极强，但非常“重”，启动需分钟级，且占用大量 CPU 和内存资源。
2. 容器化 (Container)：它是操作系统级虚拟化。容器共享宿主机的内核，只打包应用及其依赖库，因此非常“轻”，秒级启动，资源损耗极低。
4.虚拟机部署和容器化部署与系统层面交互的细节？
- 虚拟机是硬件级虚拟化，通过 Hypervisor 在物理硬件上模拟出独立的 CPU、内存和操作系统内核，交互时存在指令拦截与翻译的性能损耗；
- 而容器是内核级虚拟化，直接共享宿主机的内核，利用 Linux 的 Namespace 实现进程间的视图隔离、Cgroups 实现资源配额限制，其本质是宿主机上一个受限制的进程，因此在启动速度、资源利用率和运行效率上远超虚拟机，更适合微服务化和弹性扩缩容场景。

6.对linux防火墙的了解
- Linux 防火墙是以内核 Netfilter 为过滤基石，
- 通过 iptables 的“四表五链”实现底层包过滤与 NAT 转换，
- 或利用 firewalld 的“区域（Zone）”机制实现更高层级的动态安全管理；
- 在实际运维中，它不仅是阻断非法流量的屏障，更是我构建 SNAT/DNAT 转发网关和 DMZ 安全隔离区的核心技术手段。
7.在Linux下查询端口的占用情况？查看端口的进程情况(跟端口相关的，大概这个问题)
- 我日常首选 ss -tunlp 查看全局开放情况，因为它直接调用内核的 tcp_diag 模块，性能最好；而定位具体端口问题时，我会用 lsof -i :端口号 快速找人。
8.cpu占用满了如何解决
- 先用 top 定位 PID，再用 top -Hp 定位线程。看逻辑循环，磁盘卡顿，流量激增。
9.用户态和内核态的区别？
- 用户态是受限的应用运行环境，确保程序不会搞垮系统，比如脚本/进程；
- 而内核态是高权限的核心管理环境，负责调度硬件资源，直接执行CPU命令。
- 两者通过系统调用进行交互，虽然这种机制保护了系统稳定，但频繁的切换会产生额外的 CPU 上下文切换开销，这也是我们在做底层性能调优（如优化 Redis 或 Nginx）时重点关注的瓶颈点。
10.容器之间的隔离如何实现的？
- 容器隔离是通过 Linux 内核的 Namespace （逻辑上的视图隔离，内核维护指针，进程只能看到属于这个namespace的资源）技术实现
- 进程、
- 网络
- 文件系统
- 的视图隔离（让进程以为独占系统），
- 同时利用 Cgroups 技术实现 CPU、内存等物理资源的配额限制，
- 并配合 rootfs 确保文件系统的独立性。这种“轻量级”的隔离方式，使得容器在拥有接近原生性能的同时，具备了独立的运行环境。

K8S提供了哪些默认的业务负载类型，和deploy平行的负载类型？

  • Deployment（最常用：无状态负载）

    • 地位：与其它 4 种完全平级，是 K8S 的“头等公民”。
    • 职责：管理无状态应用。它通过控制 ReplicaSet 来保证 Pod 的副本数量和版本。
    • 核心特性：支持滚动更新、版本回退、水平扩缩容。
    • 场景：Web 服务、API 接口、微服务。

  • StatefulSet（有状态负载）—— 与 Deploy 平级

    • 地位：平级。它是为了解决 Deployment 无法处理“身份一致性”问题而设计的。
    • 职责：管理有状态应用。每个 Pod 都有一个从 0 开始的固定序号和独立的持久化存储。
    • 核心特性：Pod 名字固定（如 web-0）、Pod 启动有序、存储卷与 Pod 绑定（Pod 漂移到别的机器，存储也会跟着过去）。
    • 场景：MySQL 集群、Redis 哨兵/集群、Kafka、ZooKeeper

  • DaemonSet（守护进程负载）—— 与 Deploy 平级

    • 地位：平级。它不关心“副本数”，它关心的是“覆盖率”。
    • 职责：确保集群中的每一个（或指定的）Node 上都运行一个 Pod 实例。
    • 核心特性：新节点加入时自动拉起 Pod，节点下线时自动回收
    • 场景：日志收集 Agent（如 Fluentd）、监控 Exporter（如 Node Exporter）、网络插件（如 Calico/Flannel）。

  • Job（批处理任务）—— 与 Deploy 平级

    • 地位：平级。它属于 batch/v1 API。
    • 职责：管理一次性任务。任务执行成功（退出码 0），Pod 即告完成，不再重启。
    • 核心特性：保证指定数量的 Pod 成功结束。
    • 场景：数据库脚本执行、模型训练初始数据清洗、导出报表。

  • CronJob（定时任务）—— 与 Deploy 平级

    • 地位：平级。它是 Job 的上级管理者。
    • 职责：按照 Linux Cron 表达式定时创建 Job。
    • 核心特性：周期性触发，可以设置保留历史 Job 的数量。
    • 场景：凌晨 2 点备份数据库、每小时清理一次临时文件。

ds控制器是干什么的？

  • “DaemonSet (DS) 是 Kubernetes 中专门用于基础设施类服务的负载控制器。
  • 它的核心职责是确保在集群的每一个（或指定的）节点上，都运行且仅运行一个特定的 Pod 副本。与关注副本总数的 Deployment 不同，DS 关注的是节点覆盖率：每当有新节点加入集群，DS 会自动在该节点拉起 Pod，节点移除时则自动回收

如何自定义K8S资源？

“我觉得 K8s 资源本质上是一种状态声明。比如在我的项目中，如果我需要部署一个高可用的 MySQL，我不会一个一个去拉 Pod。我会优先考虑使用 Operator 这种自定义资源。

因为标准的 Deployment 只懂‘活着’，不懂数据库的‘主从同步’。而自定义资源配上 Operator，就像是给 K8s 请了个专业的‘数据库管理员’，它懂业务逻辑，能自动帮我处理备份和故障切换。这种把运维经验代码化的能力，才是 K8s 资源设计的精髓。”
