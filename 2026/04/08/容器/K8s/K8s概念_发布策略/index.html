K8s 原生内置策略（Deployment 自带）

这是你在 YAML 文件的 spec.strategy 字段中直接定义的。

1. 重新创建 (Recreate)

  • 逻辑：先杀掉所有旧版本的 Pod（ReplicaSet V1 -> 0），等它们全部关闭后，再启动新版本的 Pod（ReplicaSet V2 -> N）。

  • 优点：简单，不存在新旧版本同时在线的问题（避免数据冲突）。

  • 缺点：有停机时间（Downtime）。

  • 适用场景：后端架构发生重大改变，或者数据库表结构不兼容，无法支持新旧并存。

2. 滚动更新 (Rolling Update) —— 默认策略

  • 逻辑：逐个替换。先起一个新 Pod，停一个旧 Pod，直到全部完成。

  • 核心参数：

    • maxSurge：更新期间允许比预期副本数多出几个 Pod（控制资源消耗）。

    • maxUnavailable：更新期间允许有几个 Pod 不可用（控制服务能力）。

  • 优点：零停机时间，平滑过渡。

  • 缺点：集群中会同时存在新旧两个版本。如果你的前端代码和后端接口不匹配，可能会导致部分请求报错。

进阶发布策略（运维高频考察）

这些策略通常需要配合 Service 或 Ingress 来手动或通过脚本控制流量。

3. 蓝绿发布 (Blue-Green Deployment)

  • 核心思想：资源换时间。

  • 逻辑：

    1. 保持现有的“蓝色”环境（V1）不动。

    2. 部署一套完全独立的“绿色”环境（V2），并完成自动化测试。

    3. 修改 Service 的 selector 或 Ingress 的后端，将流量瞬间切换到绿色环境。

  • 优点：切换极快，回滚也极快（切回蓝色即可）。

  • 缺点：极度耗费资源（需要 2 倍的硬件资源）。

4. 金丝雀/灰度发布 (Canary Release)

  • 核心思想：小规模试错。名字源于矿工用金丝雀探测瓦斯。

  • 逻辑：

    1. 先发布一个新版本的 Pod（金丝雀 Pod）。

    2. 将一小部分流量（如 5% 或特定内测用户）导入新版本。

    3. 观察监控（错误率、延迟、日志）。

    4. 如果没有问题，逐步扩大比例，直到 100% 覆盖。

  • 优点：风险最小，发现问题只影响极少数人。

  • 缺点：配置复杂，需要精细的流量管理工具。

K8s概念_服务暴露
在 K8s 的运维体系中，服务暴露（Service Exposure） 是解决“如何让外部用户访问集群内部 Pod”的核心问题。 因为 Pod 的 IP 是动态且不可靠的，所以我们需要一个持久的入口。 服务暴露的底层原理K8s 主要通过 Service 和 Ingress 两个对象实现暴露，其核心底层技术是 kube-proxy 和 iptables/IPVS。 方式 原理 适用场景 ClusterIP 默认模式。仅集群内部访问。 数据库、内部中间件。 NodePort 在每个 Node 上开启一个相同端口（30000-32767）。 简单测试、小规模服务。 LoadBalancer 配合云厂商 API 自动分配外网 IP。 生产环境、公有云部署。 Ingress 七层转发（HTTP/HTTPS），基于域名和路径。 复杂 Web 应用、多业务共享一个 IP。
